| id | string (UUID) | `runs.id` | Primary key, generated by `crypto.randomUUID()` |

**Status lifecycle:**
```
QUEUED â†’ active â†’ done/failed/cancelled
```
- `active`: Run is executing
- `blocked`: Run hit a blocking condition (FAILED_BLOCKED, merge blocked)
- `done`: Run completed successfully (DONE phase)
- `failed`: Run failed (FAILED, FAILED_TRANSIENT, FAILED_UNSAFE)
- `cancelled`: Run cancelled by user (Issue #66)

## Recommended Realtime Transport

**Decision: SSE (Server-Sent Events)**

Basis:
- Server â†’ Client only (unidirectional status updates)
- Cancel via HTTP POST (separate, simpler)
- Event frequency: < 20 events/s (rate-limited)
- Parallel clients: < 50
- Native browser support (EventSource API)
- Simpler reconnect via HTTP streaming
- Already implemented and working

**Why not WebSocket:**
- No bidirectional control requirement (cancel is HTTP POST)
- Event frequency well within SSE capabilities (< 20/s)
- WebSocket adds complexity (upgrade, reconnect, message framing)
- SSE has native browser reconnection

## Security Considerations

| Concern | Mitigation |
|---------|------------|
| Secret exposure in events | `redactSecrets()` â€” 9 patterns + key-name masking |
| Event payload size | 5000 char truncation, 20-level recursion limit |
| Rate limit abuse | 20 events/s per run (sliding window) |
| Unauthorized SSE access | Run ID must exist (404 if not found) |
| Connection lifetime | 15s heartbeat, cleanup on close |
| Demo endpoint abuse | Gated: `NODE_ENV != production` + `POSITRON_ENABLE_DEMO_LIVE=1` |
| Cancel abuse | Idempotent, only active/blocked runs |
| No auth layer | Acceptable for MVP (internal/dev tool) |

### SSE Event Types

| Event | Payload | Description |
|-------|---------|-------------|
| `initial` | `{run, events}` | Sent on connection (includes sequence numbers) |
| `run-event` | RunEventData | New event during run execution |
| `run-update` | Partial Run | Phase/status/branch update |
| `run-control` | `{action}` | Pause/resume/abort action |
| `run-complete` | `{phase, status}` | Run finished |
| `run-evidence-created` | `{artifactId, kind, summary, createdAt}` | New artifact created |
| `run-cancelled` | `{runId, phase, status, message}` | Run cancelled |
| `heartbeat` | `{timestamp}` | 15s keepalive (also raw SSE comment) |

**W3C Last-Event-ID reconnection:** Every SSE event includes the W3C-standard `id: ${sequence}\n` field. On reconnect, the browser sends `Last-Event-ID` and the server resumes from that point. The initial event includes a `reconnected: true` flag on reconnection.

**Custom `_sequence` field:** Every event also includes the `_sequence` field in JSON payload â€” monotonically incrementing per run, starting at 1. Used by the frontend for event ordering.

### Rate Limiting

- Max: **20 events/s** per run ID (sliding 1-second window)
- Excluded from limit: `initial`, `heartbeat`
- Behavior: silently dropped (events are best-effort)
- Scope: per run ID (not global)

### Cancel Endpoint Design

| Property | Detail |
|----------|--------|
| Endpoint | `POST /api/runs/:id/cancel` |
| Idempotent | âś… True â€” cancelled runs return 200, not 409 |
| Status guard | âś… Only `active` or `blocked` runs can be cancelled |
| Atomic update | âś… `WHERE status IN ('active', 'blocked')` â€” race condition proof |
| ABORT signal | âś… Sets signal for running pipeline |
| SSE broadcast | âś… `run-cancelled` event |
| Race protection | âś… Conditional DB update prevents overwriting terminal states |

### Security: Secret Redaction

Patterns applied to ALL event data before broadcast:

```
/ghp_[a-zA-Z0-9]{36}/g       â€” GitHub classic token
/gho_[a-zA-Z0-9]{36}/g       â€” GitHub OAuth token
/github_pat_[a-zA-Z0-9_]{22,}/g â€” GitHub fine-grained PAT
/sk-[a-zA-Z0-9]{32,}/g       â€” OpenAI API key
/sk-ant-[a-zA-Z0-9_-]{32,}/g â€” Anthropic API key
/AIza[0-9A-Za-z_-]{35}/g     â€” Google API key
/Bearer\s+[A-Za-z0-9_\-\.]+/g  â€” Bearer token
/xox[bpras]-[a-zA-Z0-9-]+/g  â€” Slack token
/AKIA[0-9A-Z]{16}/g          â€” AWS access key
```

Additionally, any object key matching `/token|secret|password|key|auth/i` with string value > 10 chars is fully redacted.

**Known limitation:** The key-name regex is broad â€” it may redact benign keys like `public_key` or `author`. Use a more specific whitelist in production.
