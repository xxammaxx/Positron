## Objective

Enforce `requiresAuditLog` at runtime in the Tool Gateway. When a tool definition has `requiresAuditLog: true`, a pre-execution audit entry MUST be written before the tool executes. If audit writing fails or the audit sink is unavailable, execution MUST be blocked.

---

## 1. Runtime Enforcement Design

### 1.1 Gate Placement

New Gate 9 (Audit Enforcement) is inserted between Gate 8 (Egress check) and tool handler execution, inside `GatewayService.execute()`:

```
Gate 1 â†’ Gate 2 â†’ Gate 3 â†’ Gate 4 â†’ Gate 5 â†’ Gate 6 â†’ Gate 7 â†’ Gate 8 â†’ [Gate 9: Audit] â†’ handler(call)
```

**Rationale:** All existing security gates (enabled check, schema, lookup, phase, autonomy, approval, path boundary, egress) must pass before we check audit. If any earlier gate blocks, we never reach audit (preserving sealed/default-deny priority).

### 1.2 Audit Enforcement Logic

```
IF def.requiresAuditLog === true THEN
    IF this.onAudit is null THEN
        RETURN blocked(BLOCK_REASONS.AUDIT_LOG_MISSING, "onAudit callback not configured")
    
    TRY
        evidenceId = await this.onAudit(call)
        // Store evidenceId for later use in result
    CATCH auditError
        RETURN blocked(BLOCK_REASONS.AUDIT_LOG_MISSING, auditErrorMessage)
    
    // Audit succeeded â†’ continue to tool execution
    // Pass evidenceId through to result if needed
END IF
```

### 1.3 Risk Level Consideration

Per issue requirements:
- Test 1: Write tool + requiresAuditLog â†’ blocked (no sink)
- Test 2: Destructive tool + requiresAuditLog â†’ blocked (no sink)
- Test 7: Read-only tool WITHOUT requiresAuditLog â†’ NOT blocked
- Test 8: Read-only tool WITH requiresAuditLog â†’ needs audit sink too

**Decision:** Gate on `requiresAuditLog` flag only, NOT on `riskLevel`. The tool definition author explicitly sets `requiresAuditLog: true` when they want audit enforcement. If a read tool has this flag, it gets audited too.

However, scanner warnings can recommend setting `requiresAuditLog: true` for write/destructive tools that don't have it (informational only).

### 1.4 Sealed Gateway Priority

The sealed gateway state (registry sealed, gateway disabled) is enforced by Gate 1. Gate 1 executes BEFORE Gate 9. Therefore:
- Sealed/disabled gateway blocks execution at Gate 1 â†’ never reaches Gate 9
- Audit gate only applies when all previous gates have passed
- **Sealed/default-deny remains strictly stronger than audit gate**

## 2. Audit / Evidence Semantics

### 2.1 `onAudit` Callback

New public callback on `GatewayService`:

```typescript
/**
 * Optional pre-execution audit callback.
 * Called BEFORE tool execution when requiresAuditLog is true.
 * Returns an evidence event ID string.
 * Throws if audit/evidence write fails.
 */
public onAudit: ((call: ToolCall) => Promise<string>) | null = null;
```

### 2.2 Minimal Audit Entry (logged by callback implementer)

The `onAudit` callback implementer (server integration layer) should log:

- `runId` / `requestId` (from `call.runId`)
- `toolId` (from `call.toolId`)
- `riskLevel` (from tool definition)
- `requiresAuditLog` (true)
- `timestamp` (generated by callback)
- `decision` ("pre_execution_audit")
- `reason` ("audit gate passed â€” proceeding to execution")

### 2.3 Security Constraints

- **NO secrets** in audit entries
- **NO `.env` contents** in audit entries
- **NO unfiltered tool arguments** â€” callback implementer must sanitize/summarize
- **If sanitization unavailable:** Log only metadata (toolId, runId, timestamp)

### 2.4 Relationship to `onEvidence`

- `onAudit`: Pre-execution, writes audit trail BEFORE tool runs
- `onEvidence`: Post-execution, writes evidence event AFTER tool completes
- Both are independent callbacks that can be configured separately
- For #245, only `onAudit` availability determines audit enforcement

## 4. Error Behavior

| Condition | Behavior | Block Reason |
|-----------|----------|-------------|
| `requiresAuditLog: true` + no `onAudit` | BLOCKED | `AUDIT_LOG_MISSING` |
| `requiresAuditLog: true` + `onAudit` throws | BLOCKED | `AUDIT_LOG_MISSING` |
| `requiresAuditLog: true` + `onAudit` succeeds | ALLOWED (proceed to execution) | N/A |
| `requiresAuditLog: false/undefined` | NOT checked by audit gate | N/A |
| Sealed gateway (disabled) | BLOCKED at Gate 1 | `GATEWAY_DISABLED` |
| Sealed gateway (enabled) + requiresAuditLog | BLOCKED at Gate 9 if no sink | `AUDIT_LOG_MISSING` |

## 6. Test Plan

### 6.1 Unit Tests (gateway.test.ts additions)

- Audit gate blocks when `requiresAuditLog: true` and no `onAudit` callback
- Audit gate allows when `requiresAuditLog: true` and `onAudit` succeeds
- Audit gate blocks when `onAudit` throws
- Audit gate not triggered when `requiresAuditLog: false/undefined`
- Audit gate only reached after all previous gates pass
- Tool handler NOT called when audit gate blocks
